# 3.2-DJOS-YW 运维业务子系统标准操作流程

> **文档版本**：V1.0  
> **生效日期**：2026年3月  
> **编制部门**：道简网络运维部  
> **适用范围**：道简网络全体运维工程师、运维经理及管理人员

---

## 前言

### 1. 文档目的

本手册是道简网络操作系统（DJOS）运维业务子系统的标准操作规范，旨在为运维团队提供统一、可执行、可量化的作业标准。通过规范化运维组织与职责、客户服务等级标准、日常巡检维护标准、故障处理流程、变更管理规范、安全运维管理、运维工具与知识库、应急响应与业务连续性八大核心模块，确保运维服务高效有序，支撑道简网络客户满意度的持续提升。

### 2. 适用范围

本手册适用于道简网络全体运维人员（含运维工程师、高级工程师、技术专家、运维经理），覆盖以下业务场景：

- 客户IT基础设施日常运维
- 故障预防、发现、处理与复盘
- 变更规划、实施与验证
- 安全监控、漏洞修复与事件响应
- 客户服务沟通与满意度管理
- 运维工具使用与知识沉淀
- 应急响应与业务连续性保障

### 3. 核心原则

| 序号 | 原则 | 说明 |
|------|------|------|
| 1 | **客户价值导向** | 运维工作的核心是保障客户业务稳定运行 |
| 2 | **预防优于响应** | 通过主动巡检和监控预防故障发生 |
| 3 | **快速响应闭环** | 按照SLA要求快速响应，闭环管理 |
| 4 | **安全合规底线** | 安全是运维工作的生命线，任何时候不得突破 |
| 5 | **持续改进提升** | 通过复盘和知识沉淀不断优化运维质量 |

### 1.5 执行标准与要求

1. **值班在岗**：值班期间不得擅自离岗，有事须提前请假并安排替班。
2. **响应时效**：紧急问题须在SLA规定时间内响应和处理。
3. **交接完整**：值班交接须将所有遗留问题和注意事项交接清楚。
4. **能力认证**：运维人员须按能力等级要求获取相应认证后方可独立承担高等级任务。

## 模块二：客户服务等级标准

### 2.1 目的和适用范围

**目的**：建立差异化的客户服务等级体系，明确不同等级客户的运维服务标准，确保有限资源向高价值客户倾斜，同时保障全量客户的基础服务质量。

**适用范围**：客户分级定义、SLA标准制定、服务等级调整、服务目录管理等场景。

### 2.2 客户分级标准

#### 2.2.1 客户分级定义

| 客户等级 | 定义标准 | 运维资源配置 | 服务群要求 |
|----------|----------|--------------|------------|
| **战略客户** | 年合同额≥10万，或战略意义重大 | 专属运维+7×24小时 | 必须建立专属群 |
| **重要客户** | 年合同额3-10万 | 共享运维+7×12小时 | 必须建群 |
| **普通客户** | 年合同额1-3万 | 共享运维+工作日8小时 | 按需建群 |
| **低价值客户** | 年合同额<1万 | 共享运维+工作日8小时 | 无 |

#### 2.2.2 客户分级调整规则

| 调整类型 | 调整条件 | 调整审批 | 生效时间 |
|----------|----------|----------|----------|
| **升级** | 连续2年合同额增长≥30% | 运维经理+总经理 | 续约时生效 |
| **降级** | 连续1年合同额下降≥50% | 运维经理 | 续约时生效 |
| **临时升级** | 客户发生重大故障/项目上线 | 运维经理 | 事件结束 |

### 2.3 服务等级定义（SLA）

#### 2.3.1 服务等级标准

| 服务等级 | 适用客户 | 响应时间 | 解决时间 | 服务时段 |
|----------|----------|----------|----------|----------|
| **L1 紧急** | 战略客户核心系统故障 | 15分钟 | 4小时 | 7×24小时 |
| **L2 快速** | 战略/重要客户一般故障 | 30分钟 | 8小时 | 7×12小时 |
| **L3 标准** | 普通客户故障 | 2小时 | 24小时 | 工作日8小时 |
| **L4 常规** | 低价值客户/咨询类 | 4小时 | 72小时 | 工作日8小时 |

#### 2.3.2 SLA响应时效规则

| 场景 | 响应方式 | 时效要求 |
|------|----------|----------|
| **L1紧急** | 电话+短信+飞书群 | 15分钟内必须接听 |
| **L2快速** | 飞书群+电话 | 30分钟内响应 |
| **L3标准** | 飞书群/工单 | 2小时内响应 |
| **L4常规** | 工单 | 4小时内响应 |

### 2.4 服务目录

#### 2.4.1 基础服务目录

| 服务类别 | 服务内容 | 适用客户 | 服务方式 |
|----------|----------|----------|----------|
| **基础设施运维** | 服务器、网络设备、存储设备日常维护与故障处理 | 全部客户 | 远程/现场 |
| **系统软件运维** | 操作系统、数据库、中间件安装、配置、升级、故障处理 | 全部客户 | 远程/现场 |
| **应用系统运维** | 客户业务系统日常运行维护、功能故障处理、性能优化 | 签约客户 | 远程/现场 |
| **安全运维** | 安全监控、漏洞修复、安全事件响应、安全巡检 | 全部客户 | 远程 |
| **桌面运维** | PC终端、打印机、办公设备日常维护 | 签约客户 | 现场 |
| **云服务管理** | 云平台资源管理、监控、备份、灾备 | 签约客户 | 远程 |

#### 2.4.2 增值服务目录

| 服务类别 | 服务内容 | 适用客户 | 费用说明 |
|----------|----------|----------|----------|
| **系统优化** | 性能优化、安全加固、架构调整 | 全部客户 | 另行收费 |
| **定期巡检** | 月度/季度深度巡检，出具巡检报告 | 全部客户 | 含在合同内 |
| **应急演练** | 业务连续性演练、灾备演练 | 战略/重要客户 | 另行收费 |
| **技术培训** | 客户IT人员技术培训 | 全部客户 | 另行收费 |
| **驻场服务** | 运维工程师驻场服务 | 战略客户 | 另行收费 |

### 2.5 执行标准与要求

1. **SLA红线**：任何情况下不得突破L1紧急客户的SLA响应时间要求。
2. **服务目录透明**：服务目录须向客户透明，不得私下提供合同外的免费服务。
3. **分级动态调整**：客户分级每年评估一次，重大变化可申请临时调整。
4. **资源优先保障**：资源冲突时，高等级客户享有优先权。

## 模块三：日常巡检维护标准

### 3.1 目的和适用范围

**目的**：建立标准化的日常巡检维护体系，通过主动发现问题、及时处理隐患、保障系统稳定运行，从被动响应式运维向主动预防式运维转变。

**适用范围**：日常巡检、周期巡检、巡检报告编制、问题跟踪等场景。

### 3.2 巡检分类与周期

#### 3.2.1 巡检类型总览

| 巡检类型 | 周期 | 执行时间 | 执行人 | 时长 | 产出 |
|----------|------|----------|--------|------|------|
| **日常巡检** | 每日 | 9:00-10:00 | 值班工程师 | 30分钟/客户 | 巡检日志 |
| **周巡检** | 每周 | 周一 | 驻场/远程工程师 | 1小时/客户 | 周巡检报告 |
| **月度巡检** | 每月 | 每月第一周 | 高级工程师 | 2小时/客户 | 月度巡检报告 |
| **季度巡检** | 每季度 | 每季度第一周 | 技术专家 | 4小时/客户 | 季度评估报告 |

#### 3.2.2 巡检标准矩阵

| 系统类型 | 日常巡检项 | 周巡检项 | 月度巡检项 |
|----------|-----------|----------|-----------|
| **服务器** | CPU/内存/磁盘使用率 | 系统日志、安全更新 | 性能趋势、容量预测 |
| **网络设备** | 端口状态、流量 | 路由表、ACL规则 | 带宽利用率、故障分析 |
| **数据库** | 连接数、表空间 | 慢查询、死锁 | 性能优化、备份验证 |
| **存储系统** | 容量使用 | 磁盘健康、RAID状态 | 容量规划、扩容建议 |
| **安全设备** | 告警日志 | 策略有效性 | 漏洞扫描、基线核查 |

#### 3.3.2 日常巡检检查清单

| 检查项 | 检查内容 | 正常标准 | 异常处理 |
|--------|----------|----------|----------|
| **服务器CPU** | 当前CPU使用率 | <80% | 查看进程、考虑扩容 |
| **服务器内存** | 当前内存使用率 | <85% | 查看占用进程 |

| **备份任务** | 昨日备份是否成功 | 成功 | 检查原因，重新备份 |

### 3.6 执行标准与要求

1. **巡检完整性**：每次巡检须覆盖所有签约客户系统，不得遗漏。
2. **记录真实性**：巡检记录须真实反映系统状态，禁止伪造数据。
3. **问题闭环**：巡检发现的问题须录入工单系统，跟进直至关闭。
4. **报告时效**：各类巡检报告须按时发送，不得延迟。

## 模块四：故障处理流程

### 4.1 目的和适用范围

**目的**：建立标准化、高效化的故障处理流程，确保故障能够被快速响应、准确诊断、规范处理、及时解决，同时做好故障复盘和经验沉淀，防止同类问题重复发生。

**适用范围**：故障接报、故障分级、故障处理、故障升级、故障关闭、故障复盘等场景。

### 4.2 故障分级标准

#### 4.2.1 故障等级定义

| 故障等级 | 定义标准 | 影响范围 | 典型场景 |
|----------|----------|----------|----------|
| **P1 紧急** | 核心系统完全不可用 | 全公司/大量用户 | 服务器宕机、网络中断 |
| **P2 重大** | 核心功能不可用 | 主要业务无法运行 | ERP崩溃、核心数据库故障 |
| **P3 一般** | 非核心功能异常 | 部分用户/部分功能 | 某页面报错、某功能失效 |
| **P4 轻微** | 界面/体验问题 | 单个用户 | 显示异常、打印功能异常 |

#### 4.2.2 故障等级响应要求

| 故障等级 | 响应时间 | 解决时间 | 升级要求 | 汇报要求 |
|----------|----------|----------|----------|----------|
| **P1 紧急** | 15分钟内 | 4小时内 | 立即升级 | 每30分钟向客户汇报 |
| **P2 重大** | 30分钟内 | 8小时内 | 1小时未解决升级 | 每1小时汇报 |
| **P3 一般** | 2小时内 | 24小时内 | 4小时未解决升级 | 处理完成后汇报 |
| **P4 轻微** | 4小时内 | 72小时内 | 不强制升级 | 按需汇报 |

### 4.4 常见故障处理标准

#### 4.4.1 服务器宕机处理

```
发现服务器宕机
→ 立即查看监控（最后一次存活时间/告警信息）
→ 判断是硬件故障还是软件故障
    → 硬件故障：联系机房/供应商 → 更换硬件
    → 软件故障：尝试远程重启服务/服务器
→ 无法远程解决：预约现场处理
→ 恢复后检查数据完整性
→ 确认服务正常 → 通知客户
→ 记录故障原因 → 制定预防措施
```

#### 4.4.2 网络故障处理

```
发现网络故障
→ 判断影响范围（单个设备/整个网络）
→ 检查网络设备状态（路由器/交换机/防火墙）
→ 检查端口和网线连接
→ 重启相关网络设备（如需要）
→ 逐步排查故障点
→ 解决故障点 → 验证网络恢复
→ 检查监控是否正常 → 通知客户
→ 记录故障原因
```

#### 4.4.3 数据库故障处理

```
发现数据库异常
→ 检查数据库服务状态
→ 检查连接数、锁等待、死锁情况
→ 检查表空间使用率
→ 检查日志文件（错误日志/慢查询日志）
→ 根据错误信息判断原因
→ 采取相应措施（杀进程/扩容/优化SQL）
→ 验证数据库恢复正常
→ 检查数据完整性
→ 通知客户 → 记录原因
```

### 4.5 故障升级路径

#### 4.5.1 升级标准

| 升级触发条件 | 升级方向 | 说明 |
|--------------|----------|------|
| 初步处理无法解决 | L1 → L2（高级工程师） | -- |
| 2小时未定位根因 | L2 → L3（技术专家） | -- |
| 4小时未解决 | L3 → 运维经理 | -- |
| 影响客户核心业务 | 运维经理 → 总经理 | 立即升级 |

#### 4.5.2 升级沟通规范

| 升级场景 | 沟通内容 | 沟通对象 |
|----------|----------|----------|
| **向高级工程师升级** | 故障现象、已排查步骤、初步判断 | 高级工程师 |
| **向技术专家升级** | 完整故障背景、已尝试方案、需要的技术支持 | 技术专家 |
| **向运维经理升级** | 故障影响、当前进展、需要的管理决策 | 运维经理 |
| **向客户升级** | 故障原因、当前状态、预计解决时间 | 客户接口人/管理层 |

### 4.6 故障复盘规范

#### 4.6.1 复盘触发条件

| 复盘类型 | 触发条件 | 复盘时间 | 参与人 |
|----------|----------|----------|--------|
| **P1故障复盘** | P1级故障发生 | 48小时内 | 全员 |
| **P2故障复盘** | P2级故障发生 | 72小时内 | 相关人员 |
| **批量故障复盘** | 单日P2+故障≥3起 | 1周内 | 运维全员 |

### 4.7 执行标准与要求

1. **响应时效**：故障须在SLA规定时间内响应，超时视为服务事故。
2. **升级及时性**：达到升级条件须立即升级，不得隐瞒或拖延。
3. **沟通闭环**：处理过程中须定期向客户汇报进展，结束时须客户确认。
4. **复盘强制**：P1/P2故障必须完成复盘，未复盘不得关闭工单。
5. **知识沉淀**：所有故障处理后须更新知识库，防止同类问题重复发生。

## 模块五：变更管理规范

### 5.1 目的和适用范围

**目的**：建立规范的变更管理流程，确保所有变更经过评估、审批、实施、验证，最大限度降低变更风险，保障系统稳定运行。

**适用范围**：变更申请、变更评估、变更审批、变更实施、变更验证等场景。

### 5.2 变更分类标准

#### 5.2.1 变更类型定义

| 变更类型 | 定义 | 典型场景 | 审批层级 |
|----------|------|----------|----------|
| **标准变更** | 例行性操作，有成熟方案 | 系统补丁更新、常规备份 | 服务负责人 |
| **紧急变更** | 必须立即实施的变更 | 故障抢修、安全漏洞修复 | 口头通知，事后补批 |
| **重大变更** | 涉及核心系统，影响范围大 | 核心系统升级、网络架构调整 | 总经理审批 |

#### 5.2.2 变更影响评估

|----------|----------|----------|

| **技术风险** | 技术实现的复杂度和风险 | 高/中/低 |
| **回滚难度** | 回滚操作的复杂度和耗时 | 容易/一般/复杂 |
| **影响范围** | 受影响的用户/系统数量 | 多/中/少 |

5. **验证闭环**：变更后须验证效果，确认正常后方可关闭变更单。

## 模块六：安全运维管理

### 6.1 目的和适用范围

**目的**：建立全面的安全运维管理体系，确保客户系统安全稳定运行，及时发现和响应安全威胁，满足合规要求，保障客户数据安全。

**适用范围**：安全监控、安全巡检、漏洞管理、安全事件响应、安全加固等场景。

### 6.2 安全监控标准

| 监控项 | 监控内容 | 告警阈值 | 告警方式 | 检查频率 |
|--------|----------|----------|----------|----------|
| **入侵检测** | 异常登录、异常访问 | 1次告警 | 电话+短信 | 实时 |
| **漏洞扫描** | 系统漏洞、应用漏洞 | 高危漏洞即告警 | 飞书+邮件 | 每月 |
| **病毒查杀** | 木马、病毒、恶意软件 | 发现即告警 | 电话+短信 | 每日 |
| **密码策略** | 弱密码、过期密码 | 发现即告警 | 飞书 | 每周 |
| **配置基线** | 安全配置合规性 | 违规即告警 | 飞书 | 每周 |

### 6.3 漏洞管理

#### 6.3.1 漏洞分级标准

| 漏洞等级 | 定义 | 修复时限 | 处理方式 |
|----------|------|----------|----------|
| **高危** | 可直接被利用，导致系统沦陷 | 24小时内 | 立即修复 |
| **中危** | 需要特定条件才能利用 | 7天内 | 计划修复 |
| **低危** | 利用难度大，影响有限 | 30天内 | 定期修复 |

#### 6.3.2 漏洞处理流程

```
发现漏洞
→ 评估漏洞影响和修复风险
→ 高危漏洞：立即评估修复方案 → 紧急修复 → 验证
→ 中危漏洞：纳入周计划 → 定期修复 → 验证
→ 低危漏洞：纳入月计划 → 定期修复 → 验证
→ 更新漏洞管理台账
```

### 6.4 安全事件响应

#### 6.4.1 安全事件分级

| 事件等级 | 定义 | 响应时间 | 升级要求 |
|----------|------|----------|----------|
| **S1 严重** | 大规模数据泄露/核心系统沦陷 | 立即 | 总经理+公安机关 |
| **S2 高危** | 重要系统被入侵/数据篡改 | 15分钟内 | 运维经理+总经理 |
| **S3 中危** | 尝试入侵/可疑行为 | 1小时内 | 高级工程师跟进 |
| **S4 低危** | 扫描行为/低风险告警 | 4小时内 | 工程师处理 |

#### 6.4.2 安全事件响应流程

```
发现安全事件
→ 评估事件等级
→ S1/S2事件：立即上报总经理 → 启动应急响应
→ 隔离受影响系统 → 防止扩散
→ 定位攻击源 → 分析入侵路径
→ 清除威胁 → 修复漏洞
→ 恢复系统正常 → 确认安全
→ 编写安全事件报告 → 制定预防措施
```

### 6.5 执行标准与要求

1. **安全第一**：发现安全事件须立即响应，S1/S2事件须立即上报。
2. **漏洞不过夜**：高危漏洞须在24小时内修复，不得拖延。
3. **日志留存**：安全事件相关日志须留存至少6个月。
4. **定期巡检**：每月须完成漏洞扫描和安全评估。

## 模块七：运维工具与知识库

### 7.1 目的和适用范围

**目的**：建立统一的运维工具平台和知识管理体系，确保运维工作规范化、可追溯、可复制，通过知识沉淀不断提升团队整体能力。

**适用范围**：运维工具使用、知识库建设与更新、运维经验沉淀等场景。

### 7.2 运维工具清单

#### 7.2.1 运维工具矩阵

| 工具类别 | 工具名称 | 用途 | 使用人员 |
|----------|----------|------|----------|
| **监控工具** | Zabbix/Prometheus | 系统监控、告警 | 全体运维 |
| **日志工具** | ELK/ Loki | 日志收集分析 | 全体运维 |
| **工单系统** | 飞书工单/自研系统 | 工单管理 | 全体运维 |
| **远程工具** | 向日葵/TV | 远程协助 | 全体运维 |
| **脚本工具** | Ansible/SaltStack | 批量操作 | 高级工程师+ |
| **配置管理** | Git/Docker | 配置管理、容器化 | 高级工程师+ |
| **知识库** | 飞书云文档 | 知识沉淀 | 全体运维 |

#### 7.2.2 工具使用规范

| 工具 | 使用规范 | 账号管理 |
|------|----------|----------|
| **监控系统** | 所有人须知晓自己负责系统的监控配置 | 个人账号，不得共享 |
| **工单系统** | 所有运维工作须录入工单 | 个人账号，完整填写 |
| **知识库** | 重要操作/故障处理后须更新知识库 | 公共账号，分类管理 |
| **密码管理** | 所有系统密码存入密码管理器 | 加密存储，禁止明文 |

### 7.3 知识库管理

#### 7.3.1 知识库分类

| 知识类别 | 内容 | 更新要求 |
|----------|------|----------|
| **操作手册** | 标准化操作步骤 | 按系统变更更新 |
| **故障案例** | 故障处理过程和经验 | 重大故障处理后24小时内 |
| **常见问题** | FAQ汇总 | 发现问题时更新 |
| **技术方案** | 解决方案和技术文档 | 按需更新 |
| **培训教材** | 新人培训材料 | 每季度更新 |

#### 7.3.2 知识库更新流程

```
发现知识缺口/完成重大故障处理
→ 判断知识类型（手册/案例/FAQ/方案）
→ 编写/更新文档
→ 运维经理审核
→ 发布到知识库
→ 通知相关人员
```

### 7.4 执行标准与要求

1. **工单全覆盖**：所有运维工作须录入工单系统，口头请求不算。
2. **知识及时更新**：重大故障处理后须在24小时内完成知识沉淀。
3. **工具使用规范**：按规范使用相应工具，禁止私搭环境。
4. **密码安全**：密码不得泄露，定期更换。

## 模块八：应急响应与业务连续性

### 8.1 目的和适用范围

**目的**：建立完善的应急响应体系，确保在发生重大突发事件时能够快速、有序地响应，保障客户业务连续性，最大限度降低损失。

**适用范围**：应急响应预案制定、应急演练、重大事件处置、业务连续性保障等场景。

### 8.2 应急响应组织

#### 8.2.1 应急响应组织架构

| 角色 | 人员 | 职责 |
|------|------|------|
| **应急总指挥** | 总经理/运维经理 | 重大决策、资源调度、对外沟通 |
| **技术负责人** | 高级工程师/技术专家 | 技术方案制定与执行 |
| **协调人** | 运维工程师 | 资源协调、进度跟踪 |
| **客户沟通人** | 运维经理/销售 | 客户沟通、状态通报 |

#### 8.2.2 应急响应联络表

| 联络场景 | 联络人 | 联系方式 | 时效要求 |
|----------|--------|----------|----------|
| **P1故障升级** | 运维经理 | 手机 | 立即 |
| **重大事件上报** | 总经理 | 手机 | 立即 |
| **厂商技术支持** | 各设备厂商 | 400电话 | 按合同 |
| **机房紧急联络** | 各合作机房 | 值班电话 | 立即 |

### 8.3 应急预案

#### 8.3.1 常见应急预案场景

| 预案类型 | 触发条件 | 响应要求 |
|----------|----------|----------|
| **数据中心级故障** | 单机房整体不可用 | 立即启动灾备切换 |
| **勒索病毒攻击** | 发现勒索软件/文件被加密 | 立即隔离，停止蔓延 |
| **核心系统崩溃** | 核心业务系统完全不可用 | 15分钟内启动应急 |
| **大规模数据泄露** | 发现数据批量外泄 | 立即切断源，评估范围 |

#### 8.3.2 数据中心故障应急预案

```
发现数据中心故障
→ 判断影响范围（单机房/多机房）
→ 确认是否有灾备能力
    → 有灾备：启动灾备切换
    → 无灾备：联系机房确认恢复时间
→ 通知运维经理和总经理
→ 启动客户沟通机制
→ 每15分钟更新状态
→ 恢复后验证业务
→ 完成事件报告
```

### 8.4 业务连续性保障

#### 8.4.1 业务连续性关键指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **RTO** | ≤4小时 | 恢复时间目标 |
| **RPO** | ≤1小时 | 恢复点目标 |
| **可用率** | ≥99.9% | 年度可用率 |

#### 8.4.2 备份策略

| 数据类型 | 备份频率 | 保留周期 | 存放位置 |
|----------|----------|----------|----------|
| **核心数据库** | 每小时增量+每日全量 | 30天 | 本地+异地 |
| **配置文件** | 变更时 | 永久 | Git仓库 |
| **应用数据** | 每日 | 7天 | 本地 |
| **日志数据** | 实时同步 | 90天 | 日志服务器 |

### 8.5 应急演练

#### 8.5.1 演练类型与周期

| 演练类型 | 周期 | 参与人 | 演练内容 |
|----------|------|--------|----------|
| **桌面推演** | 每季度 | 运维全员 | 模拟场景讨论 |
| **实际演练** | 每半年 | 运维全员+客户 | 实际切换操作 |
| **预案更新** | 按事件触发 | 运维经理 | 预案修订 |

#### 8.5.2 演练记录

| 演练项目 | 演练日期 | 参与人 | 发现问题 | 改进措施 |
|----------|----------|--------|----------|----------|
| | | | | |

### 8.6 执行标准与要求

1. **预案完备**：每个重大风险场景须有对应预案并定期更新。
2. **演练常态**：每半年须完成一次实际应急演练。
3. **沟通及时**：重大事件须按要求及时通知相关方。
4. **复盘改进**：每次重大事件后须更新预案，防止同类事件。
