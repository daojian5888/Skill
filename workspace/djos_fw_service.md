# 3.1-DJOS-FW 服务子系统标准操作流程

> **文档版本**：V1.0  
> **生效日期**：2026年3月  
> **编制部门**：道简网络运维服务部  
> **适用范围**：IT全周期运维服务全业务线

---

## 前言

### 1. 文档目的

本手册是道简网络操作系统（DJOS）服务子系统的标准操作规范，旨在为IT运维服务团队提供统一、可执行、可量化的作业标准。通过规范化服务目录管理、日常巡检维护、故障管理、安全管理、变更管理、服务报告、客户服务及持续改进八大核心模块，确保运维服务质量符合企业客户的业务连续性要求，并支撑道简网络运维服务品牌的专业化建设。

### 2. 适用范围

本手册适用于道简网络全体运维服务人员（含驻场工程师、远程支持工程师及项目管理人员），覆盖以下服务场景：

- 企业客户IT基础设施运维（服务器、网络设备、存储设备）
- 桌面终端支持服务（办公设备、软件派发、故障处理）
- 信息安全监控与响应服务
- 系统变更与升级实施服务
- 客户服务满意度管理与服务质量评估

### 3. 核心原则

| 序号 | 原则 | 说明 |
|------|------|------|
| 1 | **客户优先** | 所有服务行为以保障客户业务连续性为第一优先级 |
| 2 | **标准化执行** | 严格按照既定流程执行各类运维操作，减少人为失误 |
| 3 | **可追溯性** | 全部操作留有记录，支持事后审计与复盘 |
| 4 | **持续改进** | 通过数据驱动与服务复盘，不断优化服务流程 |
| 5 | **安全合规** | 严格遵守信息安全规范，防止数据泄露与系统风险 |

## 模块一：服务目录管理

### 1.1 目的和适用范围

**目的**：建立统一的IT运维服务目录，明确各类服务的范围边界、服务等级定义（SLA）及交付标准，消除服务歧义，为客户提供清晰的服务承诺。

**适用范围**：运维服务售前咨询、合同签订、服务交付启动、服务变更评审等全生命周期环节。

### 1.2 服务目录结构

道简网络IT运维服务目录按服务层次划分为三级：

```
IT运维服务目录
├── 一级：服务大类
│   ├── 基础设施运维服务
│   ├── 桌面终端支持服务
│   ├── 安全服务
│   └── 咨询服务
├── 二级：服务子类
│   ├── 例如：基础设施运维服务 → 服务器运维、网络运维、存储运维
│   └── 例如：桌面终端支持服务 → 日常支持、资产管理、软件派发
└── 三级：服务项（具体服务条目）
    ├── 单次服务（如：一次网络故障排查）
    └── 周期服务（如：月度巡检报告）
```

### 1.3 服务等级定义（SLA）

#### 1.3.1 服务等级分类

| 服务等级 | 适用场景 | 响应时间 | 解决时间 | 服务时段 |
|----------|----------|----------|----------|----------|
| **L1 关键业务** | 核心业务系统宕机，影响全员使用 | ≤15分钟 | ≤4小时 | 7×24小时 |
| **L2 重要业务** | 部门级业务受损，影响部分用户 | ≤30分钟 | ≤8小时 | 7×12小时 |
| **L3 普通业务** | 单个用户或非关键系统故障 | ≤4小时 | ≤24小时 | 工作日5×8小时 |
| **L4 咨询优化** | 咨询类、预防性优化需求 | ≤8小时 | 按计划 | 工作日5×8小时 |

#### 1.3.2 响应时间与解决时间定义

- **响应时间**：从客户提出服务请求并获得客服/工程师确认的时间，到工程师首次响应客户的时间。
- **解决时间**：从首次响应开始，到问题被永久解决（而非临时规避）的时间。
- **暂停计时**：若问题解决需要等待客户方提供信息、审批决策或第三方配合，可暂停计时，并在工单中注明暂停原因及预计恢复时间。

### 1.5 服务范围边界（Scope）

#### 1.5.1 服务范围内

- 合同约定的IT基础设施及已纳入监控的设备
- 已登记的终端设备及标准化软件
- 运维管理平台及监控工具
- 客户书面接纳的变更请求范围内

#### 1.5.2 服务范围外（需单独报价）

- 非标准软件（未经我方安装或测试）的故障处理
- 客户自行开发的应用程序
- 网络线路运营商侧故障（广域网/互联网线路）
- 不可抗力导致的系统中断
- 超过合同约定数量的终端设备
- 重大基础设施改造/新建项目

### 1.6 执行标准与要求

1. **服务目录公示**：每个客户项目启动时，应向客户交付《服务目录清单》作为合同附件，并在项目启动会上进行宣贯。
2. **SLA承诺变更**：若客户提出超出SLA范围的服务请求，须经项目经理评估并重新报价后方可执行。
3. **服务目录更新**：每年至少一次全面评审服务目录，根据客户业务变化和技术演进调整服务项及SLA定义。
4. **服务边界确认**：项目启动时，通过《服务范围确认书》（模板见附件）明确服务边界，避免后续争议。

## 模块二：日常巡检维护

### 2.1 目的和适用范围

**目的**：通过周期性、系统化的巡检维护，及时发现IT系统潜在风险与隐患，变被动响应为主动预防，降低系统故障率，保障客户业务系统的稳定运行。

**适用范围**：所有签约运维服务项目的日常预防性维护活动，包括服务器、网络设备、存储设备、数据库、中间件及安全设备的周期性检查。

### 2.2 巡检标准

#### 2.2.1 巡检周期定义

| 巡检类型 | 周期 | 执行时机 | 执行人员 |
|----------|------|----------|----------|
| **日常巡检** | 每日 | 工作日 9:00 前完成 | 运维工程师 |
| **周巡检** | 每周 | 每周一或节假日后首个工作日 | 运维工程师 |
| **月度巡检** | 每月 | 每月第三个工作周 | 高级工程师/项目经理 |
| **季度巡检** | 每季度 | 每季度首月第二周 | 项目经理+技术专家 |

#### 2.2.2 巡检项目标准表

**（1）服务器巡检标准**

| 巡检项目 | 检查内容 | 正常指标 | 异常处理 |
|----------|----------|----------|----------|
| CPU使用率 | 峰值/平均值/进程分布 | 持续峰值≤85% | 超阈值告警→分析进程→必要时扩容 |
| 内存使用率 | 已用/可用/交换区 | 使用率≤80% | 超阈值→检查内存泄漏→建议扩容 |
| 磁盘使用率 | 各分区容量 | 系统盘≤85%，数据盘≤90% | 超阈值→清理或扩容 |
| 磁盘IO | 读写延迟/IOPS | IO等待≤15% | 异常→检查磁盘健康→排查IO瓶颈 |
| 系统日志 | /var/log/messages | 无ERROR/FATAL | 发现ERROR→记录→分析根因 |
| 服务状态 | 关键服务运行状态 | 全部运行中 | 服务停止→立即重启→记录 |
| 网络连接 | 服务器网络连通性 | 全网段可达 | 不通→排查网络配置或链路 |

**（2）网络设备巡检标准**

| 巡检项目 | 检查内容 | 正常指标 | 异常处理 |
|----------|----------|----------|----------|
| 端口状态 | 关键端口UP/DOWN状态 | 关键端口全部UP | 端口DOWN→检查链路→更换模块/线缆 |
| 带宽利用率 | 各接口流量/带宽比 | ≤70%峰值 | 超70%→分析流量来源→优化或扩容 |
| CPU/内存 | 设备资源使用率 | CPU≤70%，内存≤75% | 超阈值→检查攻击或异常会话 |
| 路由表 | 路由条目完整性 | 与基线一致 | 条目异常→核实→回滚配置 |
| 日志审计 | 登录日志/配置变更 | 无未授权访问 | 发现异常→安全事件流程 |

**（3）存储设备巡检标准**

| 巡检项目 | 检查内容 | 正常指标 | 异常处理 |
|----------|----------|----------|----------|
| 存储容量 | 已用/总容量/快照 | 使用率≤85% | 超阈值→扩容评估或清理快照 |
| 硬盘健康 | 磁盘SMART状态 | 无坏道/ Predictive Failure | 发现故障盘→立即更换→重建RAID |
| 数据保护 | 备份任务执行状态 | 最近一次备份成功 | 备份失败→立即重跑→排查原因 |
| LUN状态 | LUN映射/路径冗余 | 路径全冗余 | 路径丢失→检查HBA卡/交换机 |

### 2.5 执行标准与要求

1. **巡检真实性**：所有巡检数据必须来自实际检查结果，严禁编造数据或跳过检查项目。
2. **时效性**：日常巡检须在上午9:00前完成，周报须在每周一12:00前提交，月报须在每月第三个工作日前提交。
3. **异常上报**：发现L2级及以上异常，须在30分钟内电话通知客户IT负责人，并在工单中记录。
4. **报告存档**：所有巡检报告须在24小时内上传至项目管理平台，原始记录保留不少于12个月。
5. **交接规范**：工程师休假或调离时，须完成巡检工作完整交接，接替人员须重新确认设备清单及监控阈值配置。

## 模块三：故障管理流程

### 3.1 目的和适用范围

**目的**：建立标准化的故障处理流程，明确故障分级、响应时效、处理规范及升级机制，确保故障能够在最短时间内得到有效解决，最大限度降低对客户业务的影响。

**适用范围**：所有运维服务项目中发生的IT系统故障，包括硬件故障、软件故障、网络故障、安全事件等各类服务中断或性能下降事件。

### 3.2 故障分级标准

#### 3.2.1 故障等级定义

| 等级 | 名称 | 定义标准 | 影响范围 |
|------|------|----------|----------|
| **P1** | 紧急故障 | 系统完全不可用；核心业务中断；数据丢失风险 | 全员或多个部门无法使用核心业务 |
| **P2** | 严重故障 | 系统严重性能下降；核心功能部分不可用 | 部门级业务受损；影响50%以上用户 |
| **P3** | 一般故障 | 非核心功能故障；单用户问题；可临时规避 | 单个用户或非关键功能受影响 |
| **P4** | 轻微故障 | 咨询类请求；优化建议；非功能性需求 | 不影响业务运作 |

#### 3.2.2 故障定级参考表

| 故障场景 | 建议等级 | 说明 |
|----------|----------|------|
| 核心业务服务器宕机 | P1 | 立即处理 |

### 3.4 故障升级机制

#### 3.4.1 升级触发条件

以下任一条件满足时，须立即触发升级：

| 触发条件 | 升级说明 |
|----------|----------|
| 30分钟内无法定位故障原因 | 升级至高级工程师 |
| 需更换硬件且本地无备件 | 升级至项目经理+协调备件 |
| 故障影响范围扩大 | 升级至项目经理+安全事件需升级至管理层 |
| 超过SLA解决时间的50%仍未解决 | 升级至项目经理 |
| 涉及多系统联动问题 | 升级至技术经理或厂商支持 |

#### 3.4.2 升级路径

```
Level 1: 值班工程师/一线支持
    |
    | 升级条件触发
    v
Level 2: 高级工程师/二线支持
    |
    | 升级条件触发
    v
Level 3: 技术经理/三线支持
    |
    | 升级条件触发
    v
Level 4: 厂商技术支持（T3）
    |
    | 升级条件触发
    v
Level 5: 原厂专家/供应商
```

#### 3.4.3 升级沟通规范

- **升级通知**：升级须同时通知上一级处理人和项目经理，邮件/电话双渠道。
- **升级报告**：升级时须提供《故障处理升级报告》，包含：故障现象、已排查步骤、已采取的措施、当前状态、需要的支持。
- **升级闭环**：升级处理完成后，须由原处理人完成后续验证和工单闭环，不得遗漏。

### 3.5 故障处理时效要求

| 故障等级 | 响应时间 | 解决时间 | 超时升级 |
|----------|----------|----------|----------|
| P1 紧急 | ≤15分钟 | ≤4小时 | 超2小时未解决->升级至总经理 |
| P2 严重 | ≤30分钟 | ≤8小时 | 超4小时未解决->升级至项目经理 |
| P3 一般 | ≤4小时 | ≤24小时 | 超12小时未解决->升级至技术经理 |
| P4 轻微 | ≤8小时 | 按计划 | 按常规流程 |

### 3.6 故障记录与复盘要求

#### 3.6.1 故障工单必填字段

| 字段 | 说明 |
|------|------|
| 故障编号 | 系统自动生成，格式：INC-YYYYMMDD-XXX |
| 故障级别 | P1/P2/P3/P4 |
| 发现时间 | 精确到分钟 |
| 影响范围 | 受影响系统/用户数 |
| 故障现象 | 详细描述故障表现 |
| 处理过程 | 时间线形式记录每一步操作 |
| 根因分析 | 5-Why分析或鱼骨图分析 |
| 解决方案 | 永久性解决方案（非临时规避） |
| 预防措施 | 避免同类故障再次发生的措施 |
| 处理耗时 | 从接报到关闭的总时长 |

#### 3.6.2 复盘机制

| 故障等级 | 复盘要求 | 复盘时间 |
|----------|----------|----------|
| P1故障 | 必须召开复盘会议，输出《故障复盘报告》 | 故障关闭后48小时内 |
| P2故障 | 技术经理主持复盘，输出复盘记录 | 故障关闭后72小时内 |
| P3故障 | 工程师自行复盘，工单中填写反思总结 | 故障关闭后1周内 |
| P4故障 | 纳入月度汇总分析 | 月度 |

### 3.7 执行标准与要求

1. **第一时间响应**：所有故障报修须在SLA响应时间内首次响应，不允许无声无息。
2. **进度主动更新**：处理中故障须每30分钟向客户更新一次处理进度（如未解决）。
3. **禁止无记录关闭**：工单未经完整填写不得关闭，知识库字段必须填写完整。
4. **根因分析**：P1/P2故障必须完成5-Why根因分析，禁止仅填写"已解决"而无根因。
5. **临时方案标注**：如需临时规避故障，须在工单中明确标注"临时方案"及后续整改计划。
6. **P1故障升级**：P1故障须在15分钟内通知项目经理及客户IT负责人，每小时汇报进展。

### 4.8 执行标准与要求

1. **7x24监控**：P1/P2级服务项目须提供7x24小时安全监控，其他项目提供工作时段监控。

6. **安全保密**：安全相关信息须严格保密，禁止向无关人员透露客户安全配置、漏洞信息及事件详情。
7. **年度渗透测试**：每年至少组织一次针对客户核心系统的渗透测试，输出渗透测试报告。

## 模块五：变更管理流程

### 5.1 目的和适用范围

**目的**：建立标准化的变更管理流程，确保所有对IT环境的变更都经过评估、审批、实施和验证，在保障业务稳定性的前提下，有序推进系统优化与升级。

**适用范围**：所有涉及生产环境的配置变更、版本升级、结构调整、策略修改等操作，包括紧急变更和常规变更。

### 5.2 变更分类

| 变更类别 | 说明 | 典型场景 | 审批层级 |
|----------|------|----------|----------|
| **标准变更** | 预定义的标准操作，有成熟流程 | 定期补丁更新、标准化巡检操作 | 值班工程师即可执行（项目经理备案） |

| **紧急变更** | 突发故障必须立即执行的变更 | 紧急漏洞修复、故障临时修复 | 口头审批（48小时内补齐文档） |

### 5.5 变更审批流程

#### 5.5.1 标准变更审批

```
申请人提交 -> 项目经理
备案 -> 执行（24小时内反馈结果）
```

#### 5.5.2 常规变更审批

```
申请人提交 -> 技术经理初审 -> 项目经理审批 -> 客户负责人审批 -> 执行
（1个工作日内完成审批，紧急情况可加急）
```

#### 5.5.3 紧急变更审批

```
口头授权 -> 立即执行 -> 24小时内补办变更手续 -> 归档
（紧急变更可先执行后审批，但须在24小时内完成补批流程）
```

### 5.6 变更实施规范

1. **变更前准备**：
   - 提前2个工作日通知相关人员（客户IT负责人、业务使用方）
   - 准备好回滚所需的备份和数据
   - 确认监控和告警已就位

2. **变更实施**：
   - 严格按照审批通过的方案执行，不得擅自增加变更内容
   - 重大变更须两人在场，一人操作一人监督
   - 全程做好操作记录，精确到分钟

3. **变更后验证**：
   - 完成后立即进行功能验证
   - 观察30分钟以上确认系统稳定
   - 通知相关人员变更已完成

### 5.7 变更失败处理

| 阶段 | 处理方式 |
|------|----------|
| 变更实施中 | 立即停止操作，启动回滚方案 |
| 变更完成后功能异常 | 立即启动回滚，恢复至变更前状态 |
| 变更完成后业务异常 | 分析原因，48小时内给出解决方案 |

### 5.8 执行标准与要求

1. **禁止擅自变更**：所有生产环境变更必须经过审批，禁止工程师私自操作。
2. **回滚方案强制要求**：每项变更必须包含可执行的回滚方案，无回滚方案的变更不予审批。
3. **变更窗口**：重大变更原则上在业务低峰期（夜间/周末）执行，并在变更前通知客户。
4. **变更记录完整**：所有变更须记录操作日志，包括操作时间、操作人、操作内容、每一步的结果。
5. **紧急变更补批**：紧急变更执行后须在24小时内补齐变更申请和审批记录。
6. **变更成功率考核**：每月统计变更成功率（成功变更数/总变更数），纳入工程师绩效考核。

## 模块七：客户服务标准

### 7.1 目的和适用范围

**目的**：建立标准化的客户服务行为规范，确保运维团队以专业、友好、高效的服务态度与客户沟通，提升客户体验和满意度，塑造道简网络专业服务品牌。

**适用范围**：所有运维服务人员的日常客户服务行为，包括现场服务、远程支持、电话沟通、邮件往来等所有与客户的交互场景。

### 7.2 服务行为规范

#### 7.2.1 服务态度标准

| 场景 | 规范要求 |
|------|----------|
| **响应态度** | 任何客户请求都应积极响应，不推诿、不敷衍、不冷嘲热讽 |
| **情绪管理** | 无论客户情绪如何，始终保持专业、冷静、友好的态度 |
| **语言规范** | 使用文明用语，不使用不专业语言，不与客户争论 |
| **尊重客户** | 尊重客户的工作方式和习惯，不强制要求客户配合 |
| **主动服务** | 主动发现问题和机会，不等客户要求才行动 |

#### 7.2.2 服务形象标准

| 场景 | 规范要求 |
|------|----------|
| **着装规范** | 驻场工程师须着工作服，佩戴工牌，保持整洁得体 |
| **现场行为** | 不在客户现场饮食、吸烟、闲聊，不动用客户私人物品 |
| **沟通礼仪** | 初次见面主动问候和自我介绍，离开时告知并致谢 |
| **时间观念** | 准时到达约定地点，变更时间须提前通知并道歉 |

### 7.3 沟通标准

### 7.4 满意度管理

#### 7.4.1 满意度调查机制

| 调查类型 | 调查频率 | 调查方式 | 调查对象 |
|----------|----------|----------|----------|

| **月度满意度** | 每月末 | 在线问卷 | 客户IT负责人 |
| **季度满意度** | 每季度末 | 在线问卷+电话访谈 | 客户管理层 |

#### 7.4.2 满意度评分标准

| 评分 | 等级 | 定义 |
|------|------|------|
| 5分 | 非常满意 | 服务超出预期，完全满足需求 |
| 4分 | 满意 | 服务满足需求，体验良好 |
| 3分 | 一般 | 服务基本满足，有小瑕疵 |
| 2分 | 不满意 | 服务有较大不足 |
| 1分 | 非常不满意 | 服务有严重问题 |

#### 7.4.3 满意度改进机制

- **满意度低于4分**：须在48小时内由项目经理主动回访客户，了解不满原因，制定改进计划
- **满意度低于3分**：须在24小时内提交《客户投诉处理报告》，由技术经理及以上层级跟进
- **连续2个月满意度低于4分**：纳入重点客户管理，项目经理升级为区域经理介入
- **满意度分析**：每季度分析满意度分布及变化趋势，制定系统性改进措施

### 7.5 执行标准与要求

1. **服务禁语**：严禁使用"不知道"、"不归我管"、"你自己解决"、"这很简单"等不专业语言。
2. **服务禁止行为**：严禁未经客户允许操作客户设备、严禁在客户面前抱怨公司或同事、严禁向客户作出超出合同的承诺。
3. **隐私保护**：严禁向第三方透露客户业务信息、系统配置、人员信息等。
4. **投诉处理**：所有客户投诉须在24小时内给出初步回复，72小时内给出处理结果。
5. **满意度挂钩**：满意度评分纳入工程师绩效考核，连续3个月满意度低于3分的工程师须接受服务规范再培训。

### 8.5 人员能力提升

#### 8.5.1 培训体系

| 培训类型 | 周期 | 内容 | 考核方式 |
|----------|------|------|----------|
| **新员工入职培训** | 入职首月 | 公司文化/服务规范/基础技能/安全意识 | 结业考试 |
| **技能进阶培训** | 每季度 | 专项技能（安全/网络/存储/数据库） | 实操考核 |
| **服务规范培训** | 每半年 | 客户服务标准/沟通话术/投诉处理 | 情景模拟 |
| **认证培训** | 每年 | 厂商认证（HCIE/CCIE/CISSP等） | 认证考试 |
| **案例分享会** | 每月 | 故障复盘分享/最佳实践交流 | 无 |

#### 8.5.2 能力评估

| 能力维度 | 评估方式 | 评估周期 |
|----------|----------|----------|
| 技术技能 | 实操考核+理论考试 | 每年 |
| 服务态度 | 客户满意度评分 | 每季度 |
| 流程执行 | 流程合规检查 | 每季度 |
| 知识贡献 | 知识库文章数量与质量 | 每季度 |

### 8.6 执行标准与要求

1. **复盘强制要求**：P1/P2故障必须100%完成复盘，复盘报告作为工单关闭的必要条件。
2. **知识库更新**：每次故障处理完成后，须在48小时内完成故障案例入库。
3. **改进跟踪**：所有改进措施须录入改进跟踪系统，设置责任人、完成时限和验证方式。
4. **改进闭环验证**：改进措施完成后须验证效果，未达预期须重新制定改进方案。
5. **季度评审**：每季度召开服务评审会，客户代表参与，评审服务质量和改进进展。
6. **持续改进目标**：年度SLA达成率目标不低于99.5%，客户满意度目标不低于4.5分。
